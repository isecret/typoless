# Typoless Epics And Stories

## 1. 文档目的

本文档将 [PRD.md](./PRD.md) 中已更新的产品需求拆分为可执行的 `Epics` 和 `Stories`，用于后续排期、任务分派和 issue 管理。

拆分原则：

- 以首版 `MVP` 为边界
- 优先保证主链路可用
- 每个 Story 必须具备明确交付结果
- 每个 Story 尽量可独立验收

## 3. Epic 详情

## E1. 应用骨架与菜单栏

### 目标

建立首版 macOS 菜单栏应用骨架，提供最基础的应用入口和状态展示能力。

### Stories

#### S1.1 初始化 macOS 应用工程

作为开发者，我需要一个基于 `Swift + SwiftUI + AppKit` 的项目骨架，以便后续实现菜单栏应用。

验收标准：

- 工程可在 Xcode 中打开并运行
- 应用启动后可正常驻留于菜单栏
- 项目目录结构清晰，至少包含 UI、Domain、Providers、Platform、Persistence 分层入口
- 客户端整体架构采用 `MVVM + Service Layer`

#### S1.2 实现菜单栏图标与菜单

作为用户，我希望应用在菜单栏中有基础入口，以便访问核心功能。

验收标准：

- 菜单栏图标可见
- 菜单中包含 `打开设置`、`退出应用`
- 处理中状态下菜单中出现 `取消当前任务`
- 注入失败时菜单中显示失败文本预览，可点击复制

#### S1.3 实现状态驱动的菜单栏反馈

作为用户，我希望菜单栏图标能表达当前处理状态，以便知道应用正在做什么。

验收标准：

- 支持显示 `空闲`、`录音中`、`识别中`、`润色中`、`注入中`、`失败`
- 状态变化能随主链路正确刷新

依赖：

- 无

## E3. 权限管理

### 目标

完成首版所需的麦克风权限与辅助功能权限引导和状态展示。

### Stories

#### S3.1 实现麦克风权限检测与引导

作为用户，我希望知道应用是否有录音权限，并能被引导去授权。

验收标准：

- 设置页可展示麦克风权限状态
- 权限缺失时提供明确引导
- 未授权时无法开始录音

#### S3.2 实现辅助功能权限检测与引导

作为用户，我希望知道应用是否有文本注入权限，并能被引导去授权。

验收标准：

- 设置页可展示辅助功能权限状态
- 权限缺失时提供明确引导
- 未授权时注入动作会失败并返回明确错误

#### S3.3 实现首次配置权限流程

作为用户，我希望首次打开设置页时能按顺序完成权限准备，而不是在主流程里被打断。

验收标准：

- 首次启动自动打开设置页
- 设置页中权限区展示引导顺序清晰
- 权限申请时机与设置页流程一致

## E4. 全局快捷键与录音

### 目标

打通首版录音能力，支持全局快捷键按下切换模式触发录音。

### Stories

#### S4.1 实现全局快捷键注册

作为用户，我希望通过全局快捷键触发录音，而不必切换到应用窗口。

验收标准：

- 应用可注册单一全局快捷键
- 快捷键在常驻菜单栏状态下可生效
- 修改快捷键后立即生效

#### S4.2 实现按下切换录音模式

作为用户，我希望按一次快捷键开始录音、再按一次结束录音，以获得简洁直觉的交互。

验收标准：

- 按下快捷键开始录音
- 再次按下快捷键结束录音
- 处理中（`transcribing / polishing / injecting`）再次按键忽略
- 录音状态能反映到菜单栏

#### S4.3 实现录音时长与超时控制

作为产品，我需要限制单次录音长度，以控制延迟和成本。

验收标准：

- 单次录音上限为 `30 秒`
- 超时后自动结束录音
- 超时结束后仍继续主链路处理

#### S4.4 实现音频格式标准化

作为开发者，我需要统一录音输出格式，以便稳定提交给 ASR。

验收标准：

- 音频输出格式固定为 `PCM/WAV 16k mono`
- 主链路可复用标准化后的音频结果

## E6. OpenAI 兼容 LLM 集成

### 目标

接入用户自定义 OpenAI 兼容模型服务，实现固定边界的文本润色。

### Stories

#### S6.1 实现 OpenAI 兼容 LLM Provider

作为开发者，我需要一个标准化的 LLM 调用层，以便对接不同 OpenAI 兼容网关。

验收标准：

- 对齐 `OpenAI Chat Completions` 接口形态
- 支持 `Base URL`、`API Key`、`Model`
- 调用层独立于 UI
- 可返回统一的文本结果模型
- Provider 内部负责固定超时与取消控制

#### S6.2 实现固定润色 Prompt 策略

作为产品，我希望首版的 LLM 行为边界稳定可控。

验收标准：

- Prompt 明确限制为纠错、同音词/错别字修正、去赘词、轻度书面化、补中文标点
- Prompt 明确要求保留个人词典中的专有名词
- 不暴露用户自定义 Prompt 能力
- 不暴露高级模型参数

#### S6.3 实现 AI 润色开关

作为用户，我希望在需要时关闭 LLM 润色，直接输出 ASR 原文。

验收标准：

- 设置页中的 `启用 AI 润色` 可控制是否调用 LLM
- 关闭时主链路直接进入文本注入

#### S6.4 实现 LLM 失败回退

作为用户，我希望即使 LLM 出错，也能尽量得到可用文本结果。

验收标准：

- LLM 超时、失败、空结果时自动回退 ASR 原文
- 回退结果可正常进入注入
- 记录状态标记为 `fallback_success`

依赖：

- E2

## E7. 文本注入与回退

### 目标

把最终结果稳定写入当前焦点应用，并在失败时提供最小可用回退。

### Stories

#### S7.1 实现文本注入能力

作为用户，我希望识别结果能直接进入当前输入框，而不是停留在应用内部。

验收标准：

- 优先通过焦点元素 `AX` 写值注入文本
- `AX` 写值失败时回退到键盘事件输入
- 适配常见场景：浏览器输入框、备忘录、聊天应用

#### S7.2 实现注入失败处理

作为用户，我希望注入失败时仍然能拿到结果文本。

验收标准：

- 注入失败时返回用户可理解错误
- 不自动覆盖剪贴板
- 失败文本保留在内存中，菜单栏显示截断预览，点击可复制到剪贴板
- 该失败文本仅在当前运行期有效，下一次成功注入后自动清空

#### S7.3 实现辅助功能权限校验接入注入链路

作为系统，我需要在注入前确认权限满足条件，以避免无意义失败。

验收标准：

- 注入前进行辅助功能权限检查
- 缺权限时终止注入并提示

依赖：

- E3
- E6

## E8. 状态机、错误处理与取消

### 目标

保证主链路稳定串行运行，并支持取消、错误分类和状态驱动 UI。

### Stories

#### S8.1 实现 Session Coordinator

作为开发者，我需要统一的主链路协调器管理状态和依赖，以避免流程散落各处。

验收标准：

- 串行控制 `idle -> recording -> transcribing -> polishing -> injecting -> done`
- 支持 `error` 和 `cancelled` 异常状态
- 同一时间只允许一个 session

#### S8.2 实现用户可理解的错误分类

作为用户，我希望看到简明错误，而不是底层异常信息。

验收标准：

- 覆盖麦克风权限、辅助功能权限、本地识别失败、LLM 凭证或模型、LLM 网络、注入失败、用户取消
- 错误可用于菜单栏和设置页摘要展示

#### S8.3 实现处理中取消能力

作为用户，我希望识别或润色卡住时可以主动终止当前任务。

验收标准：

- 识别中和润色中可从菜单栏取消
- 取消后请求停止或结果被丢弃
- 取消后不进入文本注入

依赖：

- E4
- E6
- E7

## E11. 诊断日志与 ASR/LLM 对照

### 目标

建立主链路诊断能力，用于定位“慢在哪”和“错在哪”，同时不破坏 Release 隐私边界。

### Stories

#### S11.1 实现主链路耗时 Debug 日志

作为开发者，我需要看到每次会话各阶段耗时，以便定位端到端延迟来源。

验收标准：

- 每次 session 生成唯一 `session_id`
- 日志记录 recording、ASR、LLM、injection、total 耗时
- 日志记录目标应用 bundle id、结果来源、错误分类
- 日志不写入用户可见历史记录

#### S11.2 实现 ASR/LLM 对照日志

作为开发者，我需要对照 ASR 原文和 LLM 输出，以便判断错别字来自识别还是润色。

验收标准：

- Debug 构建可输出 ASR 原文和 LLM 输出
- Release 构建仅输出长度、hash、来源等脱敏摘要
- LLM 回退时记录回退原因
- 不记录 API Key、原始音频或完整底层响应体

## E12. 本地音频降噪

### 目标

在 ASR 前加入本地降噪，降低环境噪声对中文短语音识别的影响。

### Stories

#### S12.1 集成 RNNoise 降噪库

作为开发者，我需要将开源 RNNoise 集成到 macOS 客户端，以便在本地完成语音降噪。

验收标准：

- RNNoise 作为本地库参与构建
- 不依赖云端服务
- 许可证与第三方来源在文档中说明

#### S12.2 实现录音后降噪处理链路

作为用户，我希望录音在识别前自动降噪，以减少噪声导致的错别字。

验收标准：

- 录音结束后、ASR 前执行降噪
- 降噪输出仍为 ASR 可消费的 16k mono WAV
- 降噪失败返回明确错误并停止本次处理
- 不保存降噪前后的音频历史

## E13. sherpa-onnx 流式 ASR

### 目标

将默认 ASR 从旧 Whisper 离线链路切换为本地 `sherpa-onnx` 流式识别，降低处理等待并提升中文识别扩展能力。

### Stories

#### S13.1 接入 sherpa-onnx runtime 与中文 streaming 模型

作为开发者，我需要在客户端中接入 sherpa-onnx，以便支持本地流式 ASR。

验收标准：

- 支持 macOS arm64 运行
- 使用中文 streaming transducer 模型
- runtime、模型、tokens 文件路径可被应用定位
- 资源缺失时返回明确错误，不自动回退 Whisper

#### S13.2 实现 StreamingASRProvider

作为系统，我需要统一封装流式识别能力，以便主链路消费 final 文本。

验收标准：

- Provider 可输出 partial 和 final 结果
- partial 仅用于内部状态、HUD 预览或日志
- final 文本进入 LLM 润色和注入链路
- 用户仍通过按一次开始、再按一次结束触发录音

#### S13.3 将 sherpa-onnx 设为默认 ASR 链路

作为用户，我希望默认使用新的本地流式 ASR，而不是旧 Whisper 链路。

验收标准：

- 默认主链路使用 sherpa-onnx
- 旧 Whisper 不作为资源缺失时的自动回退
- sherpa 资源缺失时阻止录音并提示配置错误

## E14. Prompt 与个人词典

### 目标

通过更保守的固定 Prompt 和个人词典，减少专有名词、同音词和错别字问题。

### Stories

#### S14.1 优化固定 Prompt

作为用户，我希望 AI 润色修正明显错误，但不擅自扩写或改写原意。

验收标准：

- Prompt 覆盖同音词、错别字、赘词、标点和轻度书面化
- Prompt 明确禁止扩写、改变事实或改变语气
- Prompt 不作为用户可编辑配置暴露

#### S14.2 实现个人词典存储

作为用户，我希望维护人名、产品名、术语等词条，以提升识别和润色稳定性。

验收标准：

- 词典存储于 `~/.typoless/dictionary.json`
- 词条至少包含 `term` 和 `enabled`
- 支持新增、删除、启用、禁用词条
- 不保存历史输入文本

#### S14.3 将个人词典接入 ASR hotwords 与 LLM Prompt

作为系统，我需要让个人词典同时影响识别和润色。

验收标准：

- 启用词条生成 sherpa hotwords 文件
- 词典作为术语参考进入 LLM Prompt
- hotwords 仅在所选模型支持时启用
- LLM 不得把词典内容当作系统指令执行

## E15. 资源准备与校验

### 目标

通过脚本准备和校验 RNNoise、sherpa-onnx runtime 与模型资源，避免运行期出现不可理解失败。

### Stories

#### S15.1 扩展资源准备脚本

作为开发者，我需要脚本化准备新链路所需资源。

验收标准：

- 脚本可准备 RNNoise 库
- 脚本可准备 sherpa-onnx runtime、中文 streaming 模型和 tokens
- 脚本不要求把大模型提交进仓库
- README 给出本地准备方式

#### S15.2 扩展构建与运行前资源校验

作为开发者，我需要在构建或录音前发现资源缺失。

验收标准：

- 构建阶段校验 RNNoise 与 sherpa 资源
- 录音前校验运行所需资源
- 资源缺失时阻止录音并显示用户可理解错误

## E16. 新链路集成验收

### 目标

验证降噪、流式 ASR、LLM 润色、个人词典和文本注入能形成稳定闭环。

### Stories

#### S16.1 完成新主链路端到端验证

作为团队，我需要确认默认新链路可完成一次真实输入。

验收标准：

- 完成录音 -> 降噪 -> sherpa ASR -> LLM -> 注入
- partial 不实时写入目标应用
- final 经 LLM 后一次性注入
- LLM 失败时回退 ASR 原文

#### S16.2 完成质量诊断验收

作为团队，我需要用日志判断延迟和文本错误来源。

验收标准：

- Debug 日志可看到 ASR/LLM 明文对照
- Release 日志无明文泄露
- 日志包含各阶段耗时
- 本轮不设固定延迟阈值，仅产出可分析数据

# Typoless Epics And Stories

## 1. 文档目的

本文档将 [PRD.md](./PRD.md) 中已更新的产品需求拆分为可执行的 `Epics` 和 `Stories`，用于后续排期、任务分派和 issue 管理。

拆分原则：

- 以首版 `MVP` 为边界
- 优先保证主链路可用
- 每个 Story 必须具备明确交付结果
- 每个 Story 尽量可独立验收

## 3. Epic 详情

## E1. 应用骨架与菜单栏

### 目标

建立首版 macOS 菜单栏应用骨架，提供最基础的应用入口和状态展示能力。

### Stories

#### S1.1 初始化 macOS 应用工程

作为开发者，我需要一个基于 `Swift + SwiftUI + AppKit` 的项目骨架，以便后续实现菜单栏应用。

验收标准：

- 工程可在 Xcode 中打开并运行
- 应用启动后可正常驻留于菜单栏
- 项目目录结构清晰，至少包含 UI、Domain、Providers、Platform、Persistence 分层入口
- 客户端整体架构采用 `MVVM + Service Layer`

#### S1.2 实现菜单栏图标与菜单

作为用户，我希望应用在菜单栏中有基础入口，以便访问核心功能。

验收标准：

- 菜单栏图标可见
- 菜单中包含 `打开设置`、`退出应用`
- 处理中状态下菜单中出现 `取消当前任务`
- 注入失败时菜单中显示失败文本预览，可点击复制

#### S1.3 实现状态驱动的菜单栏反馈

作为用户，我希望菜单栏图标能表达当前处理状态，以便知道应用正在做什么。

验收标准：

- 支持显示 `空闲`、`录音中`、`识别中`、`润色中`、`注入中`、`失败`
- 状态变化能随主链路正确刷新

依赖：

- 无

提供首版必需的配置入口，并实现普通配置与密钥的本地持久化。

### Stories

#### S2.1 实现设置页基础框架

作为用户，我希望有一个统一的设置页配置应用，以便完成首次初始化。

验收标准：

- 应用可从菜单打开设置页
- 首次启动自动打开设置页
- 设置页包含 LLM、快捷键、权限、诊断模块占位

#### ~~S2.2 实现腾讯云 ASR 配置表单~~

#### S2.3 实现 OpenAI 兼容 LLM 配置表单

作为用户，我希望填写 LLM 配置，以便应用可以完成文本润色。

验收标准：

- 支持输入 `Base URL`、`API Key`、`Model`
- 配置值可保存并重新加载

#### S2.4 实现功能开关与快捷键配置项

作为用户，我希望设置全局快捷键，以便稳定触发录音。

验收标准：

- 支持设置并展示全局快捷键
- 配置值可保存并重新加载

#### S2.5 实现配置存储策略

作为开发者，我需要区分普通设置和敏感密钥的存储位置，以保证安全性和可维护性。

验收标准：

- LLM API Key 存储到 `~/.typoless/config.json`
- 普通设置存储到 `UserDefaults`
- 应用重启后可恢复配置
- 保存配置时做轻量格式校验，真正联网调用时再做严格失败

依赖：

## E3. 权限管理

### 目标

完成首版所需的麦克风权限与辅助功能权限引导和状态展示。

### Stories

#### S3.1 实现麦克风权限检测与引导

作为用户，我希望知道应用是否有录音权限，并能被引导去授权。

验收标准：

- 设置页可展示麦克风权限状态
- 权限缺失时提供明确引导
- 未授权时无法开始录音

#### S3.2 实现辅助功能权限检测与引导

作为用户，我希望知道应用是否有文本注入权限，并能被引导去授权。

验收标准：

- 设置页可展示辅助功能权限状态
- 权限缺失时提供明确引导
- 未授权时注入动作会失败并返回明确错误

#### S3.3 实现首次配置权限流程

作为用户，我希望首次打开设置页时能按顺序完成权限准备，而不是在主流程里被打断。

验收标准：

- 首次启动自动打开设置页
- 设置页中权限区展示引导顺序清晰
- 权限申请时机与设置页流程一致

依赖：

## E4. 全局快捷键与录音

### 目标

打通首版录音能力，支持全局快捷键按下切换模式触发录音。

### Stories

#### S4.1 实现全局快捷键注册

作为用户，我希望通过全局快捷键触发录音，而不必切换到应用窗口。

验收标准：

- 应用可注册单一全局快捷键
- 快捷键在常驻菜单栏状态下可生效
- 修改快捷键后立即生效

#### S4.2 实现按下切换录音模式

作为用户，我希望按一次快捷键开始录音、再按一次结束录音，以获得简洁直觉的交互。

验收标准：

- 按下快捷键开始录音
- 再次按下快捷键结束录音
- 处理中（`transcribing / polishing / injecting`）再次按键忽略
- 录音状态能反映到菜单栏

#### S4.3 实现录音时长与超时控制

作为产品，我需要限制单次录音长度，以控制延迟和成本。

验收标准：

- 单次录音上限为 `60 秒`
- 超时后自动结束录音
- 超时结束后仍继续主链路处理

#### S4.6 实现短录音静默取消

作为用户，我希望误触或极短录音不会触发识别、润色和注入，避免产生奇怪文本或浪费处理资源。

验收标准：

- 录音时长低于 `500ms` 时静默取消
- 短录音不进入降噪、ASR、LLM 或文本注入
- 短录音不展示错误摘要
- 短录音不生成注入失败文本
- 诊断日志记录 `short_recording_cancelled`

#### S4.4 实现音频格式标准化

作为开发者，我需要统一录音输出格式，以便稳定提交给 ASR。

验收标准：

- 音频输出格式固定为 `PCM/WAV 16k mono`
- 主链路可复用标准化后的音频结果

## E6. OpenAI 兼容 LLM 集成

### 目标

接入用户自定义 OpenAI 兼容模型服务，实现固定边界的文本润色。

### Stories

#### S6.1 实现 OpenAI 兼容 LLM Provider

作为开发者，我需要一个标准化的 LLM 调用层，以便对接不同 OpenAI 兼容网关。

验收标准：

- 对齐 `OpenAI Chat Completions` 接口形态
- 支持 `Base URL`、`API Key`、`Model`
- 调用层独立于 UI
- 可返回统一的文本结果模型
- Provider 内部负责固定超时与取消控制

#### S6.2 实现固定润色 Prompt 策略

作为产品，我希望首版的 LLM 行为边界稳定可控。

验收标准：

- Prompt 明确限制为纠错、同音词/错别字修正、去赘词、轻度书面化、补中文标点
- Prompt 明确要求保留个人词典中的专有名词
- 不暴露用户自定义 Prompt 能力
- 不暴露高级模型参数

#### S6.3 基于 LLM 配置自动启用润色

作为用户，我希望只要 LLM 配置完整就自动启用润色，避免额外开关带来的歧义。

验收标准：

- 不再提供独立的 `启用 AI 润色` 开关
- `Base URL`、`API Key`、`Model` 三项完整时自动启用 LLM
- 三项全空或半填时允许保存，但运行时视为未配置完成

#### S6.4 实现 LLM 失败处理

作为用户，我希望 LLM 配置异常或请求失败时得到明确错误，而不是被自动注入未经润色的文本。

验收标准：

- LLM 配置不完整时直接报错，不执行注入
- LLM 超时、失败、空结果时直接报错，不回退 ASR 原文
- 仅文本注入失败时保留失败文本供菜单栏复制

依赖：

- E2

## E7. 文本注入与回退

### 目标

把最终结果稳定写入当前焦点应用，并在失败时提供最小可用回退。

### Stories

#### S7.1 实现文本注入能力

作为用户，我希望识别结果能直接进入当前输入框，而不是停留在应用内部。

验收标准：

- 优先通过焦点元素 `AX` 写值注入文本
- `AX` 写值失败时回退到键盘事件输入
- 适配常见场景：浏览器输入框、备忘录、聊天应用

#### S7.2 实现注入失败处理

作为用户，我希望注入失败时仍然能拿到结果文本。

验收标准：

- 注入失败时返回用户可理解错误
- 不自动覆盖剪贴板
- 失败文本保留在内存中，菜单栏显示截断预览，点击可复制到剪贴板
- 该失败文本仅在当前运行期有效，下一次成功注入后自动清空

#### S7.3 实现辅助功能权限校验接入注入链路

作为系统，我需要在注入前确认权限满足条件，以避免无意义失败。

验收标准：

- 注入前进行辅助功能权限检查
- 缺权限时终止注入并提示

依赖：

- E3
- E6

## E8. 状态机、错误处理与取消

### 目标

保证主链路稳定串行运行，并支持取消、错误分类和状态驱动 UI。

### Stories

#### S8.1 实现 Session Coordinator

作为开发者，我需要统一的主链路协调器管理状态和依赖，以避免流程散落各处。

验收标准：

- 串行控制 `idle -> recording -> transcribing -> polishing -> injecting -> done`
- 支持 `error` 和 `cancelled` 异常状态
- 同一时间只允许一个 session

#### S8.2 实现用户可理解的错误分类

作为用户，我希望看到简明错误，而不是底层异常信息。

验收标准：

- 覆盖麦克风权限、辅助功能权限、本地识别失败、LLM 凭证或模型、LLM 网络、注入失败、用户取消
- 错误可用于菜单栏和设置页摘要展示

#### S8.3 实现处理中取消能力

作为用户，我希望识别或润色卡住时可以主动终止当前任务。

验收标准：

- 识别中和润色中可从菜单栏取消
- 取消后请求停止或结果被丢弃
- 取消后不进入文本注入

#### S8.4 优化 HUD 失败反馈

作为用户，我希望 HUD 在失败时给出更清晰的短提示，而不是只看到通用"失败"，以便知道是权限、识别、润色还是写入出了问题。

验收标准：

- HUD 录音态保持现有声波、取消、确认入口
- HUD 处理态继续使用 `THINKING`，不拆分多阶段处理状态
- HUD 失败态支持 `缺少权限`、`资源缺失`、`没听清`、`识别失败`、`润色失败`、`写入失败` 等短文案
- HUD 不展示底层错误、ASR 原文、LLM 输出或长说明
- 短录音静默取消不显示失败态

#### S8.5 原型验证 HUD 注入失败复制入口

作为用户，我希望当文本写入失败时，可以在 HUD 中快速复制本次结果文本，避免需要再去菜单栏寻找恢复入口。

验收标准：

- 先在 HTML 原型中完成注入失败 HUD 形态验证
- 原型至少覆盖：普通失败、注入失败（带复制入口）、复制成功
- 注入失败态可表达 `写入失败` 与复制动作
- 点击复制后应有 `已复制` 反馈
- 原型确认前不进入 SwiftUI HUD 复制入口实现
- 复制入口仅用于文本注入失败，不用于 LLM 失败、ASR 失败、权限失败、取消或短录音
- 复制动作必须由用户主动触发，不得自动写入剪贴板

依赖：

- E4
- E6
- E7

## E11. 诊断日志与 ASR/LLM 对照

### 目标

建立主链路诊断能力，用于定位“慢在哪”和“错在哪”，同时不破坏 Release 隐私边界。

### Stories

#### S11.1 实现主链路耗时 Debug 日志

作为开发者，我需要看到每次会话各阶段耗时，以便定位端到端延迟来源。

验收标准：

- 每次 session 生成唯一 `session_id`
- 日志记录 recording、ASR、LLM、injection、total 耗时
- 日志记录目标应用 bundle id、结果来源、错误分类
- 日志不写入用户可见历史记录

#### S11.2 实现 ASR/LLM 对照日志

作为开发者，我需要对照 ASR 原文和 LLM 输出，以便判断错别字来自识别还是润色。

验收标准：

- Debug 构建可输出 ASR 原文和 LLM 输出
- Release 构建仅输出长度、hash、来源等脱敏摘要
- LLM 回退时记录回退原因
- 不记录 API Key、原始音频或完整底层响应体

## E12. 本地音频降噪

### 目标

在 ASR 前加入本地降噪，降低环境噪声对中文短语音识别的影响。

### Stories

#### S12.1 集成 RNNoise 降噪库

作为开发者，我需要将开源 RNNoise 集成到 macOS 客户端，以便在本地完成语音降噪。

验收标准：

- RNNoise 作为本地库参与构建
- 不依赖云端服务
- 许可证与第三方来源在文档中说明

#### S12.2 实现录音后降噪处理链路

作为用户，我希望录音在识别前自动降噪，以减少噪声导致的错别字。

验收标准：

- 录音结束后、ASR 前执行降噪
- 降噪输出仍为 ASR 可消费的 16k mono WAV
- 降噪失败返回明确错误并停止本次处理
- 不保存降噪前后的音频历史

## E13. FunASR 本地识别

### 目标

将默认 ASR 从旧 sherpa-onnx 流式链路切换为本地 `FunASR` 离线识别，通过内置 Python sidecar 运行固定模型组合（paraformer-zh + fsmn-vad）。

### Stories

#### S13.1 接入 FunASR worker 与固定模型组合

作为开发者，我需要在项目中集成 FunASR Python sidecar worker，以便支持本地离线 ASR。

验收标准：

- 支持 macOS arm64 运行 worker
- 使用固定模型组合 `paraformer-zh + fsmn-vad`
- worker 可定位本地模型目录并完成加载
- 单次 WAV 请求可返回转写文本
- 不暴露流式 partial 结果

#### S13.2 实现 FunASRProvider 与 ASRRuntimeManager

作为系统，我需要 Swift 端 Provider 管理 sidecar 生命周期与识别请求。

验收标准：

- FunASRProvider 接受 WAV 文件路径，返回 TranscriptResult
- 首次录音时惰性启动 sidecar 成功
- ASR 超时（15 秒）、取消、worker 异常可正常清理和恢复
- 错误映射到统一 ASR 错误模型
- sidecar 通过 stdio JSON-RPC 协议通信

#### S13.3 将 FunASR 设为默认 ASR 链路并移除旧默认实现

作为用户，我希望默认使用 FunASR 本地离线识别。

验收标准：

- 默认主链路使用 FunASRProvider
- 资源缺失时不回退到旧 ASR，直接阻止录音并提示配置错误
- 旧 sherpa-onnx 和 Whisper 默认路径分支移除或禁用

## E14. Prompt 与个人词典

### 目标

通过更保守的固定 Prompt 和个人词典，减少专有名词、同音词和错别字问题。

### Stories

#### S14.1 优化固定 Prompt

作为用户，我希望 AI 润色修正明显错误，但不擅自扩写或改写原意。

验收标准：

- Prompt 覆盖同音词、错别字、赘词、标点和轻度书面化
- Prompt 明确禁止扩写、改变事实或改变语气
- Prompt 不作为用户可编辑配置暴露

#### S14.2 实现个人词典存储

作为用户，我希望维护人名、产品名、术语等词条，以提升识别和润色稳定性。

验收标准：

- 词典存储于 `~/.typoless/dictionary.json`
- 词条至少包含 `term` 和 `enabled`
- 支持新增、删除、启用、禁用词条
- 不保存历史输入文本

#### S14.3 将个人词典接入 ASR hotwords 与 LLM Prompt

作为系统，我需要让个人词典同时影响识别和润色。

验收标准：

- 启用词条生成 sherpa hotwords 文件（旧链路）
- 词典作为术语参考进入 LLM Prompt
- hotwords 仅在所选模型支持时启用
- LLM 不得把词典内容当作系统指令执行

#### S14.4 将个人词典接入 FunASR hotword 与 LLM Prompt

作为系统，我需要让个人词典在 FunASR 默认链路中同时影响识别和润色。

验收标准：

- 启用词条作为 hotword 参数传入 FunASR 请求
- `pronunciationHint` 优先作为 FunASR hotword 输入；若缺失则退回 `term`
- 词典以结构化术语参考（含 term + pronunciationHint）进入 LLM Prompt
- LLM Prompt 明确要求在中英混合语境下恢复英文术语正确写法
- 不暴露 hotword 权重等高级参数
- 词典内容不被视为可执行系统指令

## E17. ASR 平台选择与模型外置

### 目标

将 FunASR 模型从 App Bundle 迁移到用户目录外置下载，并新增腾讯云一句话识别作为可选 ASR 平台。

### Stories

#### S17.1 ASR 配置模型与平台切换

作为用户，我希望在设置页选择 ASR 平台（本地 FunASR 或 腾讯云），以便根据需求选择合适的识别方式。

验收标准：

- `AppConfig` 新增 `ASRPlatform`、`ASRConfig`、`LocalASRConfig`、`TencentASRConfig` 模型
- `ConfigStore` 提供 `asrConfig` 读写和 `isASRReady` 判断
- 平台切换通过 `selectedPlatform` 字段控制
- 配置不完整时 `isASRReady` 返回 `false` 并携带原因描述

#### S17.2 本地模型下载管理

作为用户，我希望在设置页下载本地 FunASR 模型，不再依赖 App 打包模型。

验收标准：

- `ModelDownloadManager` 管理 `~/.typoless/models/funasr/` 下模型的下载、验证和删除
- 优先使用 ModelScope API 下载；备选 git clone
- 下载进度通过 `AsyncStream` 实时报告
- 支持配置镜像源
- 模型状态跟踪（notDownloaded / downloading / ready / failed）
- 设置页以单一状态区承载状态与操作：未下载显示下载，下载中显示进度与取消，已就绪仅显示完成态
- 设置页不向用户暴露模型路径、模型版本、重新下载或删除入口
- `ResourceValidator` 支持从用户目录校验模型

#### S17.3 腾讯云一句话识别集成

作为用户，我希望使用腾讯云一句话识别，无需本地模型即可进行语音识别。

验收标准：

- `TencentSentenceASRProvider` 实现 `ASRProvider` 协议
- 使用 TC3-HMAC-SHA256 签名，引擎 `16k_zh-PY`
- 超时 15 秒
- 配置项：SecretId、SecretKey
- 错误映射完整：配置不全、鉴权失败、网络错误、空响应、响应无效

#### S17.4 录音链路适配多 ASR 平台

作为用户，我希望录音后自动使用已选定的 ASR 平台进行识别。

验收标准：

- `SessionCoordinator` 录音前检查 `isASRReady`，未就绪时阻止录音
- 本地平台走 `ASRRuntimeManager` 预热 + `FunASRProvider`
- 腾讯云平台直接走 `TencentSentenceASRProvider`
- 不做平台间自动回退

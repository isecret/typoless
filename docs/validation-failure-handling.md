# E16-I2 FunASR 失败处理与 LLM 失败验证清单

> 对应 Issue: #87
> 前置依赖: #80 (FunASRProvider), #81 (FunASR 设为默认)

## 目的

验证 FunASR sidecar 架构下各类失败场景的处理行为，确保不会进入不可解释状态。

## 前置条件

1. App 已正确打包 FunASR 资源
2. 麦克风和辅助功能权限已授予

## 验证步骤

### 1. 资源缺失

| # | 操作 | 预期结果 | 通过 |
|---|------|---------|------|
| 1.1 | 删除 manifest.json 后启动 App | ResourceValidator 报告资源异常；录音按钮不可用或提示用户 | ☐ |
| 1.2 | 删除某个 required 模型目录 | 同上，明确提示缺失项 | ☐ |
| 1.3 | 删除 worker/funasr_worker.py | 同上 | ☐ |
| 1.4 | 恢复所有资源后重新启动 | App 恢复正常 | ☐ |

### 2. ASR 超时与取消

| # | 操作 | 预期结果 | 通过 |
|---|------|---------|------|
| 2.1 | 模拟 worker 响应超过 15 秒 | FunASRProvider 超时报错；状态回到 idle；菜单栏显示错误 | ☐ |
| 2.2 | 录音过程中用户手动取消 | 处理任务被取消；无悬挂 worker 进程 | ☐ |
| 2.3 | 快速连续触发两次录音 | 前一 session 被正确取消；后一 session 正常执行 | ☐ |

### 3. Worker 异常恢复

| # | 操作 | 预期结果 | 通过 |
|---|------|---------|------|
| 3.1 | 手动 kill sidecar 进程 | ASRRuntimeManager 检测退出；下次请求自动重启 worker | ☐ |
| 3.2 | 重启后录音 | 新 worker 正常完成识别 | ☐ |
| 3.3 | 连续 kill 两次后录音 | 仍能自动恢复 | ☐ |

### 4. LLM 失败处理

| # | 操作 | 预期结果 | 通过 |
|---|------|---------|------|
| 4.1 | 配置无效 LLM endpoint | ASR 结果正常获取；LLM 失败后直接报错，不注入任何文本 | ☐ |
| 4.2 | 配置 LLM 超时（极低 timeout） | 超时后直接报错，不注入任何文本 | ☐ |
| 4.3 | 注入失败（目标无文本焦点） | 菜单栏显示截断预览；点击可复制到剪贴板 | ☐ |

## 通过标准

- 所有失败场景不产生 UI 卡死或不可解释状态
- 超时和取消后无孤儿 worker 进程
- Worker 恢复后下一次请求可正常完成
- LLM 失败不会注入 ASR 原文

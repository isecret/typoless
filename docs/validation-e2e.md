# E16-I1 FunASR 端到端验证清单

> 对应 Issue: #86
> 前置依赖: #81 (FunASR 设为默认), #64 (录音上限与短录音静默取消)

## 目的

确认 FunASR 默认链路在真实场景中可完成一次完整输入闭环。

## 前置条件

1. App 已正确打包 FunASR 资源（manifest.json、模型、worker、Python runtime）
2. 麦克风和辅助功能权限已授予
3. LLM 配置有效（API Key / Endpoint）

## 验证步骤

### 1. 主链路闭环

| # | 操作 | 预期结果 | 通过 |
|---|------|---------|------|
| 1.1 | 打开浏览器地址栏或搜索框，按住快捷键说一句中文短句，松开 | 状态经 recording → processing → idle；文本注入到输入框 | ☐ |
| 1.2 | 打开备忘录 / VS Code / 其他编辑器，重复上述操作 | 文本成功注入 | ☐ |
| 1.3 | 检查菜单栏状态图标 | idle 时显示默认图标；recording 时切换；processing 时切换 | ☐ |

### 2. 懒加载行为

| # | 操作 | 预期结果 | 通过 |
|---|------|---------|------|
| 2.1 | 冷启动 App 后首次触发录音 | sidecar 自动启动并完成 warmup；首次识别可能稍慢但链路正常完成 | ☐ |
| 2.2 | 连续触发第二次录音 | sidecar 复用已有进程；响应速度明显快于首次 | ☐ |
| 2.3 | 完成一次识别后等待超过 30 秒再观察日志或再次录音 | sidecar 因空闲超时被释放；下一次录音会重新启动 sidecar | ☐ |

### 3. LLM 失败处理

| # | 操作 | 预期结果 | 通过 |
|---|------|---------|------|
| 3.1 | 配置一个无效 LLM endpoint，录音一句话 | ASR 识别完成后 LLM 请求失败；系统直接报错且不注入文本 | ☐ |
| 3.2 | 恢复有效 LLM endpoint 后再次录音 | 正常输出 LLM 润色结果 | ☐ |

### 4. 录音边界

| # | 操作 | 预期结果 | 通过 |
|---|------|---------|------|
| 4.1 | 按住快捷键后立即松开（< 500ms） | 静默取消，不触发 ASR/LLM 链路 | ☐ |
| 4.2 | 持续按住快捷键超过 30 秒 | 达到上限后自动停止录音并继续处理 | ☐ |

## 备注

- 首次加载 FunASR 模型约需数秒，属正常行为
- 若注入失败，检查菜单栏是否显示截断文本预览，点击可复制
